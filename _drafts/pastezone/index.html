    PasteZone

    Platform: Hackviser
    Type: Scenario
    Difficulty: Easy
    Link: PasteZone

    Description

    "An urgent midnight intelligence alert has triggered a full-scale response: sensitive data belonging to the Ministry of Energy has been discovered for sale on PasteZone, one of the most notorious illegal platforms on the dark web. Cybercriminals use this site to anonymously share details of compromised systems, vulnerabilities, and stolen data. Preliminary investigations have confirmed that the leaked content originates directly from the Ministry of Energy’s systems, posing a significant security threat.

    Your mission is to identify the individuals or groups responsible for this breach, gather intelligence on PasteZone’s administrators, and collect actionable evidence."

    Initial Enumeration

    This particular scenario gives us the domain/machine name for the target, rather than the IP address. I started by ascertaining the IP address for the target:

    Target host can be added to /etc/hosts if required but was not necessary for this scenario

    I generated a list of open ports for more comprehensive enumeration with the following: ports=$(nmap -p- --min-rate=1000 TARGET_IP_ADDRESS | grep ^[0-9] | cut -d '/' -f 1 | tr '\n' ',' | sed s/,$//)
    This revealed the following open ports:
    - 22
    - 80 (obviously, given the provided URL)

    Looking at the source code for the home page reveals little we didn't already know from the ffuf output. Navigating to each of the links in the title bar reveals functioning pages with content:

    Initial foothold

    Looking at the wording for the next task, it would seem that we need to gain access to the logs on the web server hosting the web page. With no usernames, no "secret" directories, and no secrets in the available source code, I decided to test the input forms at create.php and search.php for some basic injection attacks:
    - A basic SQLi payload (using a ') failed to leak any error messages in any of the form fields. At this early stage in the investigation I decided to shelve this technique, at least temporarily.
    - A basic XSS payload (<script>alert(1)</script>) resulted in the payload being rendered as plaintext for the forum post form and treated as a literal search string for the search form. With this in mind, I decided to shelve this technique, at least temporarily.

    Recalling the initial ffuf results, I decided to try some recursive enumeration on the discovered directories. There was nothing unexpected found in the assets search, but enumerating the vendor directory revealed something potentially interesting:

    Twig is a template engine for use with PHP web pages:

    Privilege Escalation

    I tried to upgrade my bash shell to meterpreter in order to upload an automated script such as LinPEAS, but this was unsuccessful. Attempting to copy and paste the script contents into a local copy was painfully slow, so I abandoned that and proceded to manually enumerate possibilities for privilege escalation using the following checklist:

    • OS
    • Kernel
    • Running services
    • Running processes (ps aux)
    • Installed packages and versions
    • Logged in users (ps au)
    • Contents of /home directories
    • SSH directory (keys)
    • Bash history
    • Sudo privileges (sudo -l)
    • Configuration files (.conf and .config)
    • /etc/shadow file (readable hashes)
    • /etc/passwd file (readable hashes)
    • Cron jobs (cat /etc/crontab)
    • File systems/additional drives
    • Files with SUID (+ GTFOBins)
    • Files with SGID (+ GTFOBins)
    • Writable directories
    • Writable files
    • Capabilities (getcap)
    • Shared libraries (ldd <executable>)
